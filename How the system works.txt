## What T2AutoTron Is

T2AutoTron is a home‑automation hub that lets you control and monitor all of your smart devices—lights, switches, thermostats, power meters, and more—from a single, easy‑to‑use interface. Think of it as a universal remote for everything in your house, but smarter: it not only turns things on and off, it also gathers data (like how much electricity each device is using) and shows you real‑time insights.

## How the Pieces Fit Together (Lay‑man’s View)

1. **The “Brain” (Server)** – This is the central computer that runs on your machine. It starts up, reads its configuration (like passwords for your devices), and connects to the internet and your local network. Its job is to talk to all the different device brands (Home Assistant, Hue, Kasa, Shelly, etc.) and keep everything coordinated.

2. **Plug‑in Managers** – For each brand of device there’s a small plug‑in (a self‑contained piece of code). These plug‑ins know how to speak the language of that brand, fetch the list of devices, read their status, and send commands (e.g., “turn the kitchen light on”). Because they’re modular, adding a new brand later only means dropping a new plug‑in file—no need to rewrite the core.

3. **Routes (API Endpoints)** – The server exposes simple web URLs (like `/api/lights/ha`) that the front‑end can call to get data or control devices. These are the “doors” through which the UI asks the brain for information.

4. **Real‑time Updates (Socket.IO)** – Instead of the UI constantly polling for changes, the server pushes updates instantly over a live connection. When a light is turned on via a phone app, the server immediately notifies the UI so the display stays in sync.

5. **Caching Layer** – To avoid hammering the devices with repeated requests, the server temporarily stores (caches) recent data. This makes the system faster and reduces network traffic, while still keeping the information fresh.

6. **The Front‑end (Web UI)** – This is the visual part you see in a browser or Electron window. It loads the list of devices, shows their current state, and displays a live power‑usage meter that adds up the consumption of all devices. The UI waits for the live connection before setting up its listeners, ensuring everything works smoothly.

7. **Energy Display** – A special component reads power‑usage attributes from the various devices (Kasa plugs, Shelly meters, Home Assistant sensors) and calculates total consumption and how many devices are currently active. It updates the screen in real time, giving you an at‑a‑glance view of your home’s electricity use.

**Putting It All Together**

When you start T2AutoTron, the server boots, loads all the plug‑ins, and discovers every smart device you have. It then serves the web UI, which connects back to the server via a live socket. From that moment:

- The UI shows a list of lights, switches, and power‑metered devices.
- Any change you make in the UI (e.g., dim a light) is sent to the appropriate plug‑in, which talks to the device.
- The device’s response (or any external change, like a voice command) is pushed back instantly to the UI, keeping everything in sync.
- The energy display continuously aggregates power data, so you can see how much electricity is being used right now.

In short, T2AutoTron ties together many different smart‑home brands into one cohesive, responsive system that’s easy for anyone to use—no programming knowledge required.

## T2AutoTron System Overview (as of 2025‑11‑20)

**Server Startup**
- Loads environment variables from `.env`.
- Initializes Express, HTTP server, and Socket.IO with CORS and security settings.
- Connects to MongoDB, displays a banner via `figlet` and `chalk`.
- Calls `initializeDeviceService()` which loads device managers and sets up routes.

**Plugin Loader (`src/devices/pluginLoader.js`)**
- Dynamically loads all manager files in `src/devices/managers` (HomeAssistant, Hue, Kasa, Shelly, etc.).
- Normalizes each manager to a common interface: `initialize`, `controlDevice`, `getDevices`, `shutdown`.
- Registers API route files from `src/api/routes` under `/api/<type>/<prefix>`.

**Device Service (`src/devices/services/deviceService.js`)**
- Aggregates the loaded managers into a unified service.
- Provides methods to initialize devices, control them, fetch state, and list all devices.
- Emits a `notificationEmitter` for UI‑side notifications.

**HomeAssistant Manager (`src/devices/managers/homeAssistantManager.js`)**
- Implements performance caching: `stateCache` (5 s TTL) and `deviceCache` (30 s TTL).
- Extracts full device attributes, including power usage, and includes them in emitted states.
- WebSocket connection streams real‑time state changes and invalidates caches on updates.

**Socket Handlers (`src/api/socketHandlers.js`)**
- Registers Socket.IO events: `device-state-update`, `device-list-update`, `weather-update`, `forecast-update`, and log forwarding.
- Emits unified device list with power data after each manager initialization.

**Frontend (`src/frontend`)**
- Serves static files and custom LiteGraph nodes via `/api/custom-nodes`.
- `nodeLoader.js` dynamically imports node scripts listed by the backend.
- UI initialization now waits for the Socket.IO connection before attaching listeners.

**Energy Display (`src/frontend/js/energy-display.js`)**
- Central `getPowerFromDevice` extracts power usage from Kasa plugs, Shelly devices, and Home Assistant entities.
- Calculates total power consumption and active device count, updating the UI in real time.

**Caching Strategy**
- State cache TTL: 5 seconds.
- Device list cache TTL: 30 seconds.
- Cache invalidation occurs on state change events and after control actions.

**Logging**
- Central `logger` writes structured logs with levels (`info`, `warn`, `error`).
- Socket.IO errors and client connections are logged.

**Security & Configuration**
- Sensitive values stored in `.env` and never committed.
- CSP and CORS middleware protect the API.
- HA token endpoint masks the token in responses.

**Overall Flow**
1. Server starts → loads env, creates Express & Socket.IO.
2. `pluginLoader` loads managers & routes.
3. `DeviceService` initializes managers, registers routes, and sets up notifications.
4. Managers fetch devices, populate caches, and emit initial `device-list-update`.
5. Frontend connects via Socket.IO, receives device list, weather, and power data.
6. UI components (e.g., `energy-display.js`) render real‑time information.
